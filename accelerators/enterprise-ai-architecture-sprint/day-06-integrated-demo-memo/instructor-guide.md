# Instructor Guide — Day 6: Integrated Demo Memo

## 1. Teaching Goal

Students learn to package a bounded demo as architecture evidence.

## 2. Pre-Class Diagnostic

1. What does the demo prove?
2. What does the demo not yet prove?
3. Which component latency matters most?
4. Which audit fields prove the workflow happened?
5. What acceptance criterion would a reviewer use?

## 3. 150-Minute Flow

| Time | Activity | Output |
|---:|---|---|
| 0-10 | Diagnostic | Scope risks |
| 10-35 | Demo flow | Demo script |
| 35-60 | Latency table | Measurement plan |
| 60-90 | Architecture memo | Layer outline |
| 90-115 | Known limitations | Scope controls |
| 115-140 | Acceptance checklist | Reviewable gate |
| 140-150 | Handoff | Onboarding inputs |

## 4. Failure Gallery

### Failure 1: Demo without measurement

Looks impressive but has no latency or quality evidence.

### Failure 2: Production claim from prototype

Fix by naming current scope and next validation layer.

### Failure 3: Architecture memo lists tools only

Fix by organizing around audio, knowledge, agent, governance, deployment, and
validation layers.

### Failure 4: No acceptance criteria

Fix by naming functional, quality, latency, security, and ops evidence.

## 5. TA Workflow

1. Read demo script.
2. Check memo layers.
3. Inspect latency table.
4. Check known limitations for positive scope-control language.
5. Grade acceptance checklist.
